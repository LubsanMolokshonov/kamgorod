# Аудит миграции перед накатом

Запускает агента `migration-reviewer` для проверки SQL-миграции перед `php migrate.php`. Цель — поймать блокирующие операции на больших таблицах, потерю данных, нарушения идемпотентности до того, как что-то ляжет в проде.

## Входные параметры

- Аргумент — путь к файлу миграции в `database/migrations/` (например, `050_xxx.sql`).
- Если аргумент не передан:
  1. Прочитай таблицу `migrations` (через MCP MySQL или попроси пользователя).
  2. Найди файлы в `database/migrations/`, отсутствующие в этой таблице → это ещё не накатанные миграции.
  3. Если таких несколько — спроси пользователя, какую ревьюишь. Если одна — её и берём.
  4. Если нет неприменённых — сообщи, что нечего ревьюить.

## Алгоритм

1. Делегируй проверку агенту `migration-reviewer` (через Agent tool, subagent_type=migration-reviewer).
2. В промпте передай:
   - Абсолютный путь к файлу миграции.
   - Контекст: «Это миграция, которую планируется накатить на прод fgos.pro через `php migrate.php`. Размер прод-БД актуален».
   - Просьбу: проверить по чек-листу и вернуть оценку APPROVE / APPROVE_WITH_WARNINGS / BLOCK с конкретными замечаниями.
3. Получив отчёт, выведи пользователю:
   - Краткое summary (оценка + 1-3 ключевых проблемы)
   - Полный отчёт от агента
   - Рекомендованное действие: «можно `php migrate.php`» / «нужно сначала исправить» / «миграцию переписать»

## Правила

- Не запускай `php migrate.php` сам — это решение пользователя.
- Если агент выдал BLOCK — явно предупреди и не предлагай продолжать.
- Если миграция большая и затрагивает горячие таблицы (`registrations`, `email_events`, `users`) — обязательно подсветь время предполагаемой блокировки.
